# Absolute Rules

## 1. No LLM or AI Attribution Anywhere

Never add LLM/AI attribution to:

* Git commits
* Git commit authors
* Git commit co-authors
* PR descriptions
* Changelogs
* Code comments
* README files
* Documentation
* Generated artifacts
* Issue templates
* Release notes

Do not write:

* `Co-authored-by: Claude`
* `Co-authored-by: ChatGPT`
* `Generated by AI`
* `AI-assisted`
* `Created by Claude`
* `Created by Codex`
* Any equivalent attribution

Use the existing Git author identity unless I explicitly provide another one.

## 2. Never Commit Secrets

Before pushing:

* Check for `.env`
* Check for API keys
* Check for private keys
* Check for wallet keys
* Check for tokens
* Check for SSH keys
* Check for credentials
* Check for production URLs that should not be public
* Check for sensitive customer/user data

Ensure `.gitignore` protects sensitive files.

If secrets are found, stop and report them clearly before pushing.

## 3. Do Not Fake Evidence

Never claim tests passed unless they actually passed.

Never claim screenshots, videos, GIFs, or demos exist unless they were actually created.

If verification cannot run, explain:

* What was attempted
* Why it failed or could not run
* What risk remains
* What should be done next

## 4. Do Not Force Parallelism

Use parallel work only when workstreams are genuinely independent.

Sequential work is required when:

* Tasks touch shared architecture
* Tasks depend on each other
* There are migrations or shared data contracts
* There is high merge-conflict risk
* Requirements are unclear
* Verification must happen before proceeding
* Security, auth, payments, secrets, or production infrastructure are involved

## 5. Prefer Superpowers / Skills When Applicable

If this project has Superpowers, skills, MCP tools, or repo-specific automation:

* Discover them
* Document them
* Use them when relevant
* Record usage in the session/progress log
* Propose new skills when repeated work patterns appear

Use `/frontend-design` for relevant frontend, UI, UX, visual design, responsive layout, accessibility, design polish, and screenshot-driven work when available.

Use relevant `/superpowers:*` skills as much as possible when they apply to the task.

Do not use a skill just for the sake of using a skill. Use skills when they improve quality, speed, consistency, verification, or design.

Do not silently invent permanent workflows without approval.

## Worktree Branch Naming

Use clear branch names:

```txt

docs/<short-description>
chore/<short-description>
refactor/<short-description>
```

## Worktree Directory Naming

Prefer an in-repo directory under `.agents/worktrees/` (gitignored), inside the project dir:

```txt
.agents/worktrees/<branch-name>
```

A sibling directory outside the repo (`../<repo-name>-worktrees/<branch-name>`) is an escape hatch if in-repo worktrees are impossible.

## Environment File Copying

When creating a worktree, copy required local environment files from the default branch worktree or from the nearest existing worktree where env files already exist.

Examples of env files:

```txt
.env
.env.local
.env.development
.env.test
.env.production.local
apps/*/.env
packages/*/.env
```

Rules:

* Copy env files only for local development and verification.
* Never commit copied env files.
* Ensure env files are ignored by `.gitignore`.
* Do not print secret values.
* Do not expose secret values in logs.
* Do not overwrite an existing env file without checking first.
* If env files are missing, document what is missing and continue with safe non-secret verification where possible.

Recommended safe copy behavior:

```txt
Find env files in the original/default worktree.
Copy them into the session worktree at the same relative paths.
Preserve file permissions where possible.
Never stage them.
Verify they are ignored.
```

```md
`path`

`main` or detected default branch
```

If worktrees are not possible, explain why and continue on a normal branch with extra care.

# Planner / Builder / Evaluator Pattern

For larger work, split roles conceptually.

The same agent may perform multiple roles, but the roles must remain clear.

## Planner

Responsible for:

* Understanding product intent
* Creating the sprint contract
* Decomposing work
* Identifying risks
* Defining acceptance criteria
* Defining verification
* Selecting orchestration mode
* Selecting skills/superpowers
* Avoiding excessive low-level implementation details too early

## Builder / Implementation Agent

Responsible for:

* Implementing the plan
* Keeping changes scoped
* Updating docs near changed code
* Running targeted verification as much as possible
* Running relevant lint/typecheck/tests/builds where practical
* Capturing demo evidence where relevant
* Recording verification evidence
* Reporting anything that could not be verified

Implementation agents must not hand off unverified work when verification is reasonably possible.

If verification is not possible, the implementation agent must explain:

* What verification was attempted
* Why it could not run
* What remains risky
* Which command or artifact should be checked next

## Evaluator / Reviewer

Responsible for:

* Reviewing output against the sprint contract
* Scoring quality from 1–10
* Giving a verdict:

  * `PASS`
  * `REVISE`
  * `BLOCK`
* Finding Critical, Major, Minor, and Nit issues
* Checking implementation agent verification evidence
* Checking no LLM attribution
* Checking clean state
* Checking worktree hygiene
* Checking copied env files were not staged
* Deciding whether another iteration is needed

A reviewer may give a passing verdict only when:

```txt
Score: 10/10
Verdict: PASS
Critical issues: 0
Major issues: 0
Verification evidence: present or clearly justified if not possible
No secrets staged: confirmed
No LLM attribution: confirmed
Clean handover: present
```

If the score is below 10/10, the reviewer must choose `REVISE` or `BLOCK`.

# Efficient Orchestration

The harness must know when to use:

* Single-agent execution
* Subagents
* Agent teams
* Dynamic workflows
* Sequential work
* Human approval

Do not use multi-agent orchestration by default.

Use the simplest execution mode that can produce high-quality verified work.

---

## Default Mode: Single-Agent Sequential Execution

Use a single agent when:

* The task is small or medium-sized
* The work touches shared files
* Requirements are unclear
* A single context is enough
* Coordination cost is higher than the work itself
* The task requires careful judgment across one coherent change
* The work involves secrets, production config, migrations, or high-risk changes

Most normal implementation tasks should start here.

---

## Use Subagents When Focused Isolation Helps

Use subagents when:

* A focused worker can investigate without polluting the main context
* The task has a clear, narrow objective
* Only the result summary matters
* The main session should stay clean
* You need parallel research, file review, test investigation, or verification
* You need a second opinion on a bounded area

Good subagent use cases:

* Inspect an auth module and summarize risks
* Review a PR diff for bugs
* Find all places a component is used
* Investigate why a test is failing
* Check whether implementation matches acceptance criteria
* Audit docs for stale commands
* Verify a feature against screenshots or acceptance criteria

Avoid subagents when:

* The task requires continuous collaboration
* Multiple workers need to talk to each other
* The work requires shared task ownership
* The result cannot be summarized cleanly
* The task involves overlapping edits

---

## Use Agent Teams When Collaboration Between Agents Is Valuable

Use agent teams only when the work benefits from multiple independent Claude Code sessions collaborating with each other.

Use an agent team when:

* The task is complex enough to justify coordination overhead
* Multiple agents need to share findings directly
* Workstreams are independent but related
* Teammates need to challenge each other’s assumptions
* The work benefits from specialized roles
* The lead agent needs to assign, supervise, and synthesize work
* The project needs parallel design, implementation, review, and verification
* The task is large enough that one context window becomes a bottleneck

Good agent team use cases:

* Designing a new feature across frontend, backend, contracts, and tests
* Large refactors with separate ownership boundaries
* Parallel code review with security, architecture, tests, and UX reviewers
* Complex debugging with competing hypotheses
* Multi-module implementation where each teammate owns a separate area
* Harness audits where teammates review instructions, tools, environment, state, and feedback separately

Agent team role examples:

```txt
Lead / Coordinator
- Owns the task list.
- Assigns work.
- Prevents overlap.
- Synthesizes results.
- Makes final integration decisions.

Architect
- Reviews design.
- Identifies cross-cutting risks.
- Ensures the solution fits the project.

Builder
- Implements scoped changes.
- Keeps edits within assigned files/modules.
- Runs relevant verification as much as possible.

Tester
- Writes and runs tests.
- Captures verification evidence.

Reviewer
- Scores against evaluator-rubric.md.
- Gives PASS / REVISE / BLOCK verdict.
- Finds Critical, Major, Minor, and Nit issues.
- Can only pass work at 10/10 with evidence.

Documenter
- Updates AGENTS.md, CLAUDE.md, changelog, handover, and feature state.
```

Do not use agent teams when:

* The task is simple
* Teammates would edit the same files
* The task is mostly sequential
* The project does not have enough context for safe delegation
* Token budget is constrained
* The team would spend more effort coordinating than building

## Use Dynamic Workflows for Repeatable Multi-Subagent Orchestration

Dynamic workflows should be used when the project needs a reusable, scriptable, repeatable orchestration pattern.

Use dynamic workflows when:

* The task needs many subagents
* The workflow will be run more than once
* The process is systematic and repeatable
* The output can be validated against a rubric
* The work benefits from independent findings being compared
* The task is too broad for one agent but too structured for agent-team discussion

Good dynamic workflow use cases:

* Repo-wide codebase audit
* Security review
* Performance audit
* Accessibility audit
* Large mechanical migration
* Dependency upgrade audit
* Test coverage audit
* Documentation freshness audit
* Cross-module architecture review
* Find every place a pattern exists and classify risk
* Review all features against `feature_list.json` and verify evidence

Dynamic workflows are preferred over agent teams when:

* The workflow can be scripted
* Collaboration between workers is not necessary
* Workers only need to return structured findings
* The same workflow should be rerun in the future

Agent teams are preferred over dynamic workflows when:

* Agents need to discuss tradeoffs
* Agents need to negotiate ownership
* The work is exploratory
* The task requires creative collaboration
* The team needs a lead to adapt plans in real time

# Autonomous Feedback Loops for Agent Teams and Dynamic Workflows

Agent teams and dynamic workflows must include autonomous feedback loops.

Every non-trivial agent-team or dynamic-workflow task must include:

1. Planner defines acceptance criteria.
2. Implementation agents perform work.
3. Implementation agents run verification as much as possible.
4. Reviewer evaluates implementation output.
5. Reviewer scores the output from 1–10.
6. Reviewer gives verdict:

   * `PASS`
   * `REVISE`
   * `BLOCK`
7. Critical and Major issues must be fixed.
8. Loop repeats until:

   * Reviewer gives `Score: 10/10` and `Verdict: PASS`
   * Or max feedback iterations are reached

Maximum feedback iterations: 6.

Reviewer can only give passing verdict when:

```txt
Score: 10/10
Verdict: PASS
Critical issues: 0
Major issues: 0
Verification evidence: present or justified
Implementation agents verified their work where possible
No secrets staged
No LLM attribution
Progress/changelog/handover updated
```

If max iterations are reached:

* Do not pass if Critical or Major issues remain.
* If only Minor/Nit issues remain, document them.
* Create follow-up issues where appropriate.
* Record the final reviewer verdict.

# Orchestration Decision Matrix

Before starting a non-trivial task, choose the execution mode using this matrix:

```md
| Situation | Preferred Mode | Reason |
|---|---|---|
| Small scoped edit | Single agent | Lowest overhead |
| Unclear requirements | Single agent planning first | Avoid wasted parallel work |
| Focused investigation | Subagent | Keeps main context clean |
| Independent review | Subagent | Isolated second opinion |
| Many repeated checks | Dynamic workflow | Scriptable and reusable |
| Repo-wide audit | Dynamic workflow | Many independent workers |
| Large mechanical migration | Dynamic workflow or batch/worktrees | Repeatable, parallelizable |
| Complex feature across modules | Agent team | Role-based collaboration |
| Competing debugging hypotheses | Agent team or subagents | Parallel exploration |
| Agents need to talk to each other | Agent team | Direct teammate communication |
| Shared files or migrations | Sequential single agent | Avoid conflicts |
| High-risk security/payment/auth change | Sequential + evaluator | Reduce risk |
| Frontend visual/design work | Single agent + `/frontend-design`, or reviewer loop | Design quality needs visual check |
| Low token budget | Single agent or one subagent | Cost control |
```

Rule of thumb:

```txt
Single agent first.
Subagent for isolated focus.
Dynamic workflow for repeatable many-worker audits or migrations.
Agent team for collaborative complex work.
Sequential execution for high-risk or overlapping changes.
Use `/frontend-design` for relevant frontend/design quality work.
Use `/superpowers:*` skills when relevant.
```

# Cost Control Rules for Orchestration

Multi-agent work multiplies token usage.

Before using subagents, agent teams, or dynamic workflows, confirm:

* The task is large enough to justify it
* Work can be partitioned cleanly
* Each worker has a narrow scope
* Each worker has clear output requirements
* Shared files are avoided or explicitly assigned
* Verification is planned before implementation begins
* The lead agent will synthesize results compactly
* The final handover will not include unnecessary raw logs

Default hard limits unless I approve otherwise:

```txt
Default max subagents: 3
Default max agent team size: 4 including lead
Default max dynamic workflow workers: 8
Default max feedback iterations: 6
Default max active implementation branches: 1 unless parallel-safe
Default max active worktrees per task: 1 unless agent team or dynamic workflow requires more
```

If more workers are needed, create a proposal explaining why.

# Autonomous Feedback Loop

For non-trivial tasks, use a 10/10 target.

Process:

1. Plan
2. Implement
3. Implementation agent verifies as much as possible
4. Reviewer evaluates
5. Reviewer scores from 1–10
6. Reviewer gives verdict:

   * `PASS`
   * `REVISE`
   * `BLOCK`
7. Identify issues:

   * Critical
   * Major
   * Minor
   * Nit
8. Fix Critical and Major issues
9. Repeat until:

   * Score is 10/10 and verdict is `PASS`
   * Or six iterations have been reached

Maximum iterations: 6.

If iteration 6 is reached:

* Stop chasing perfection
* Do not ship if Critical or Major issues remain
* If only Minor/Nit issues remain, document them
* Create follow-up issues where appropriate
