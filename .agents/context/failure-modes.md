# Harness Failure Modes

How the harness itself fails, and how to recover. This is reference doctrine — the *catalogue* of what goes wrong and the standard recovery. Actual incidents are recorded in [`../logs/failures.md`](../logs/failures.md); recurring environment/tooling failures go in [`troubleshooting.md`](troubleshooting.md).

Each failure is tagged by the subsystem it belongs to: **Instructions · Tools · Environment · State · Feedback** (plus cross-cutting **Orchestration · Worktrees**).

---

## State failures (most common)

| Failure | Symptom | Recovery | Prevention |
|---|---|---|---|
| **Stale progress** | New session can't tell what's done/next | Reconstruct from git log + logs; rewrite `progress.md` | Update `progress.md` + handover every session (`/final-handoff`) |
| **Lost handover** | Next agent re-discovers context | Read `decisions.md` + `changelog.md` to rebuild | Always write `session-handoff.md` before stopping |
| **Context drift** | An `AGENTS.md` describes files that moved/vanished | `/refresh-context <dir>` | Update context maps in the same change that moves files |
| **feature_list pollution** (template repo) | Template ships with a project's real features | Reset to example entries | Track template-repo work in logs, not `feature_list.json` |

## Feedback failures

| Failure | Symptom | Recovery | Prevention |
|---|---|---|---|
| **Premature victory** | "Done" with no evidence | Re-open; run verification; capture evidence | Definition of Done gate; reviewer checks evidence |
| **Rubber-stamp review** | PASS without inspecting the diff | Re-review against the rubric, adversarially | Strict 10/10 mode; [`opus-code-review`](../../.claude/skills/opus-code-review/SKILL.md) |
| **Pass-at-9** in a loop | Loop exits below 10/10 | Resume the loop; fix Major issues | `/review-loop` enforces strict 10/10 |
| **Agreement-as-proof** | Multiple agents "agree", no checks run | Run the actual checks | Never treat consensus as verification |

## Orchestration failures

| Failure | Symptom | Recovery | Prevention |
|---|---|---|---|
| **Unbounded loop** | A loop never stops | Kill it; apply the iteration cap | Every loop has a hard cap ([`../workflows/autonomous-loops.md`](../workflows/autonomous-loops.md)) |
| **Forced parallelism** | Parallel agents edit shared files → conflicts | Serialize; re-partition by file ownership | Parallelization assessment before fan-out |
| **Over-orchestration** | Agent team for a one-line fix | Drop to a single agent | Orchestration matrix; lightest mode that works |
| **Lost worker output** | Workers produce essays, no synthesis | Re-run with structured outputs + a synthesis step | Define output schema up front |

## Tools failures

| Failure | Symptom | Recovery | Prevention |
|---|---|---|---|
| **Stray command file** | A non-command `.md` in `.claude/commands/` registers as a command | Move it to `.agents/context/`; remove from commands dir | Namespace rule (see [`slash-commands.md`](slash-commands.md)); `verify-harness.sh` check |
| **Schema drift** | A command/skill missing required headings/frontmatter | Fix to schema; re-run verify | `verify-harness.sh` schema checks fail CI |
| **Skill name ≠ dir** | Skill not discovered | Rename so `name` == directory | Verify-harness skill check |
| **False-positive scans** | Verify fails on `node_modules`/build output | Scope the scan globs | Single-sourced `EXCLUDE_DIRS` prunes vendored/build/worktree paths — **resolved & regression-tested** by `scripts/test-verify-harness.sh` (issue #5) |

## Environment failures

| Failure | Symptom | Recovery | Prevention |
|---|---|---|---|
| **Verification can't run** | Missing runtime/parser | Document what was attempted + the risk; run what you can | `init.sh` reports missing tools; degrade with warnings |
| **Staged env file** | `.env` shows in `git status` staged | Unstage; confirm `.gitignore` covers it | Never stage copied env files; verify ignored |

## Instructions failures

| Failure | Symptom | Recovery | Prevention |
|---|---|---|---|
| **Entry-file bloat** | `AGENTS.md`/`CLAUDE.md` over the length limit | Split detail into `context/`/`workflows/` | `verify-harness.sh` length check (≤200 / ≤250) |
| **AI attribution** | "Generated by …" in a commit/doc | Remove it; amend if not pushed | `verify-harness.sh` attribution scan; reviewer gate |
| **Tribal knowledge** | A rule lives only in an agent's head | Write it into the relevant doc | Persist learnings to `logs/learnings.md` |

---

## Recovery protocol (any failure)

1. **Stop** — don't pile fixes on a broken state.
2. **Attribute** — record the failure in [`../logs/failures.md`](../logs/failures.md): context, symptom, root cause, category.
3. **Recover** — apply the standard recovery above; verify it.
4. **Harden** — if a harness gap caused it, propose a fix (`/skill-evolution-loop` or [`../workflows/workflow-improvement.md`](../workflows/workflow-improvement.md)) so it can't recur.
